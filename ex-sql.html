SQL 練習帳

このページでは SQL の基礎をガンツやクレアと一緒に学んでいきます。

ガンツはドワーフの万屋よろずや 経営者です。手先が器用なので、材料さえ仕入れられれば何でも作ってしまいます。ところが、在庫管理やお金の計算はあまり得意ではなく、毎年税金もごっそり取られてしまいます。最近、在庫管理をしっかりして、売上帳をつけたり、複式簿記で貸借対照表を作ったりすれば、税金が一部控除されることもあると知って、そのための助手を雇うことにしました。

メイドのクレアは、ガンツに雇われた少し口の悪い普通の人間です。 IT に詳しく、今回はガンツの経営の補助をします。実は会計処理についてはそんなに詳しくないため、複式簿記だのなんだのといった会計処理については知らん顔をして、もっぱらガンツの業務をデータベース化することに尽力していきます。

以下、データベース操作は SQLite を基準にします。解答欄ではユーザ入力 SQL が直接実行可能になっていますが、システムはローカル環境で完結しているため、サーバサイドへの影響は一切ありません。またユーザの進捗は、ブラウザのローカルストレージにデータを保存していくため、同一ブラウザで開けば前回の続きから実行できます。

01. ドワーフ・ガンツの万屋事始め

クレアさんよぅ、なにやらデータベースというものを使うと、たくさんのデータをとてもいい感じに整理できるらしいな。早速やってみたいんだが、何をどうしたらいいんだ？

まずはド素人のガンツさんにでも分かるように、商品管理をするためのテーブルでも作りましょう。

テーブル？ 木製か？ ワシは得意じゃぞ！

違います。商品一覧を管理する表のようなものです。

なんじゃ、ただの表かい。だったら始めからそう言ってくれ。

データベースでは表ではなく、テーブルといいます。他に列はカラム、行はレコードと呼ぶことがあります。例えば商品管理ならこんな感じです。テーブル名は今回は products とでもしておきましょう。列名は順に ID 、商品名、価格、在庫数ですね。商品名は文字列、それ以外は数値のみを扱うようにします。

products
id product_name price stock_quantity
1 桐の棒 1,200 10
2 鉄の斧 16,000 8

なんじゃい。こんなただの表をやけに大げさに扱うんじゃな。

データベースはこういった表の集まりです。これからこれを SQL と呼ばれる言語を使って操作していきます。

えすきゅーえる？

はい。 SQL は Structured Query Language の略で、直訳すると「構造化問い合わせ言語」です。砕いて言うと「ルールに従って整理されたデータを取り出すための言語」といったところです。……データベースには様々な製品がありますが、製品ごとの細かい方言のようなものを別にすれば、基本的な SQL 文はおよそ共通しています。

それを使えば、大量のデータを効率的に扱えるようになるんじゃな？

はい。

じゃあ、早速それを使ってガッポガッポ儲けたいもんじゃな。

解説 01

ここをクリックすれば解説が表示されます。

テーブルを作成するには、最低限テーブル名・列名・各列の型名を指定します。例えばこんな感じです。 

CREATE TABLE `products` (
  `id` INTEGER,
  `product_name` TEXT,
  `price` INTEGER,
  `stock_quantity` INTEGER
);

なんじゃい、型名って？

それぞれの列にはどんなデータを入るのか、予め設定しておく必要があります。整数値を入れるのか・実数値を入れるのか・文字列を入れるのか、そういうことを予め決めておくことで、人間にもデータベースシステムにもここにどんな種類の値が入るのかが分かるようになります。

ん－、急にやっかいな感じになるが、よく見るとテーブル名・列名・その型名を指定してるだけじゃな。型名というのは、ここでは TEXT と INTEGER のことなんじゃな？

はい。 SQLite には主に 5 種類のデータ型が扱われます。当分文字列型の TEXT と整数型の INTEGER しか出てこないので、これだけ覚えておけば十分です。

他に注意点はあるか？

はい。改行は可読性を上げるためのものなので、なくても大丈夫です。テーブル名や列名以外の命令文の各単語は大文字で書いていますが、これも慣習と可読性を上げるためのものなので、小文字でも問題ありません。

ほう。緩いんだな。

ただし、単語間の区切りを表すカンマ（ , ）や文の終了を表すセミコロン（ ; ）は必須です。忘れると文法エラーになります。

そこは厳しいのか。

また、列名は単語間をアンダースコア（ _ ）で連結するスネークケース（ snake_case ）で書く慣習があります。パスカルケース（ PascalCase ）やキャメルケース（ camelCase ）を使いません。列名やテーブル名はバッククォート（ ` ）またはダブルクォート（ "  ）で囲うことができます。バッククォート（ ` ）は MySQL というデータベース製品と互換性があります。バッククォート（ ` ）はシングルクォート（ ' ）と違いますよ。 Shift + @ で表示できます。なくても問題ありませんが、データベースシステムによっては列名やテーブル名を囲っていない場合、強制的に大文字になってしまうことがあります。共同開発するような場合は注意が必要です。

ややこしいな。とりあえず、ここまでの内容を整理すると、テーブルを作成するには CREATE TABLE 文を使う。列名にはスネークケースを使いバッククォート（ ` ）またはダブルクォート（ " ）で囲う。カンマ（ , ）や文の終了を表すセミコロン（ ; ）には注意、そんなところじゃな。

はい。最低限はこれだけでもいいんですけど、もう少しいくつか各列に制約をつけておきます。まず ID 列は主キー制約を課し、かつ自動で連番を付けてくれるようにします。連番は 1 からはじまります。 0 からではありません。他の列は NOT NULL 制約を付けて値を書かずに空欄のままにしておくことを禁止します。 

CREATE TABLE `products` (
  `id` INTEGER PRIMARY KEY AUTOINCREMENT,
  `product_name` TEXT NOT NULL,
  `price` INTEGER NOT NULL,
  `stock_quantity` INTEGER NOT NULL
);

主キーって何じゃ？

行（レコード）を一意に識別するための列のことです。主キーの値には、他のレコードと重複してはいけない、空値が不可という性質があります。また SQLite の場合 AUTOINCREMENT を指定しなくても、主キーは自動で連番が付されますし、主キーにふさわしく一意であれば自分で値を指定してしまうこともできます。

主キーなんかなくても、商品名欄があればいいのでは？

そうでしょうか？ 例えば「ノレイ・ヴィトンの杖」は 280,000 円なのに、なぜか同じ商品名で 2,000 円のものもあります。これはどうします？

そ、それは……へへへ。

02. ドワーフ・ガンツのテーブル情報確認

クレアさんよぅ、テーブルを作ったのはいいが、本当にテーブルができてるのかどうか、これではまるで分からんじゃろ。もしかしたら「はいできました！」と言って全然できてない無能な弟子みたいな状態かもしれん。できてなかったら、結局ワシが後で尻ぬぐいをする羽目になるだけじゃ。

ド素人にしては感が良いですね。実は IT の世界では正常終了の場合何も表示しないという場合が少なくありません。特にコマンドツールでは、正常は沈黙・異常は表示という設計が割と基本です。

じゃあ、どうやって確認するんじゃ？ 信頼するしかないのか？

とんでもありません。システムもユーザも信頼厳禁です！ 軽々しい信頼はただの怠惰です。なにかしたら確認する方法も一緒に覚えておきましょう。

解説 02

ここをクリックすれば解説が表示されます。

テーブル情報を確認するには PRAGMA 文を使います。

PRAGMA table_info(`products`);

実行するとこんな感じです。

cid name type notnull dflt_value pk
0 id INTEGER 0 1
1 product_name TEXT 1 0
2 price INTEGER 1 0
3 stock_quantity INTEGER 1 0

なんかよく分からないぞ。

まず cid は Column ID で列の番号です。 0 から始まります。今回は 4 列あるので 0 から 3 が割り当てられます。

なるほど。内部的には、列にも勝手に ID が振られてるんじゃな。

name は列名です。

これは自分が作った列名通りじゃな。

type はデータ型です。 INTEGER は整数型、 TEXT は文字列型です。

テーブル作成時に指定した通りになっているか、ここで確認できるな

とは言え、実は SQLite は型に寛容（ manifest typing ）なので、 INTEGER を指定していたはずの列に '100円' という文字列を入力できてしまいます。

便利なのか？ それとも信頼できないシステムなのか？

信頼できないので、注意が必要です。

じゃあ、何のための型指定なんじゃ？ ややこしいにも程がある。

たとえある程度の柔軟性が SQLite に許されていたとしても、他のシステム・たとえば型を厳密に扱うプログラムやデータベースと連結したいというような場合に、型指定があるとないとでは扱いの容易さが全く違ってきます。それに整数型が入るべき列に文字列型が入っていると、比較や演算で問題が出ることもあります。

なるほど。……それにそもそも manifest typing って直訳したら「明白な型」みたいな意味じゃろ？ 勝手に型が変わって何が明白なんじゃ？

列指定ではなく、値そのものに現れている型みたいな含意があるようです。

そうなのか。

続いて notnull は NOT NULL 制約の有無です。 0 なら NULL 可、 1 なら NULL 不可で空値の指定は許されません。

あっ！ おいおい、 id 列が主キー列なのに、 NULL 可になってるぞ。このテーブル壊れてんじゃないのか？

フッ、ド素人なのに、なかなか注意深いですね。この列は PRIMARY KEY 指定しているので、自動採番されます。大丈夫です。

そ、そうなのか。

続いて dflt_value はデフォルト値です。実は何も指定していない場合に自動的に入力される値を設定しておけます。今回は設定していないので、すべて空欄です。

そんな指定ができるなんて初耳だぞ。

言ってませんから。……最後の pk は主キーかどうかです。 Primary Key の頭文字でしょう。今回は id 列のみ 1 で主キーとなっています。他の列は 0 なので主キーではありません。

主キーは列ひとつしかあり得ないわけじゃな？

いいえ。複数主キーというのもあります。

ふぅん。……そのうち出てきそうじゃな。

なお PRAGMA 文は SQLite 専用で、標準 SQL ではありません。例えば MySQL や MariaDB では同じような情報を取得したい場合 SHOW 文を使います。

SHOW COLUMNS FROM `products`;

とりあえず今は PRAGMA が SQLite 固有の動作設定・メタ情報取得用のコマンドだということを覚えておけば十分です。

ん？ メタ情報とは？

メタ情報が何かは、メタ情報でないものが何なのかを考えれば分かります。テーブルに格納される商品名や価格・在庫数などはメタ情報ではありません。それに対して、テーブルのフォーマットがどうなっているか、どの列が主キーでどの列には NOT NULL 制約があるかみたいなことは「情報を扱うための情報」なのでメタ情報です。つまりテーブル・列・制約などのデータです。

なるほど。「メタ」というのは確か「～について」とか「～を超えて」みたいな意味だったな。つまりテーブルに格納されるデータそのものじゃなくて、データのためのデータ・もしくはデータについてのデータということじゃな？

そうです。「桐の棒」が情報だとすれば、「「桐の棒」が文字列である」みたいなことはメタ情報なのです。

03. ドワーフ・ガンツのデータ挿入

ようやくデータ入力じゃな。

そうです。データベースには CRUD と呼ばれる 4 つの基本操作があります。

くらっど？

はい。 CRUD は Create/Read/Update/Delete の頭文字を取ったもので、それぞれデータの挿入・取得・更新・削除を表します。

なるほど。基本操作は 4 種類しかないんじゃな。今回はデータを新しく挿入したいので、テーブルを作ったときと同様に Create を使うんじゃな？

違います。新規データの挿入は INSERT 文を使います。

だったら CRUD じゃなくて IRUD じゃろ？

いいえ。実は Read 文もなくて、実際にあるのは SELECT 文なので百歩譲っても IRUD にはなりません。

だったら ISUD じゃ！

いいえ、なかなかしつこいですね。 CRUD と呼ばれるデータ操作は、データベースだけの話ではなく、あらゆる場面を想定したデータ操作の抽象概念なのです。

というと？

例えば「移動する」という動詞は、足を使う場合の「歩く」や自転車を使う場合の「漕ぐ」や車を使う場合の「走る」などというさまざまな場面を想定した抽象概念と捉えることができます。 CRUD も同様に、データ取り扱い上の抽象概念で、データベースを扱うための SQL 文においては、 Create は INSERT 文として実装されているということなのです。

なるほど。つまりデータベース以外でデータを取り扱う場合などは、また別の実装がされてたりするんじゃな。……うん、なんか少しイライラしてきたぞ。で、データの挿入はどうするんじゃ？

では INSERT 文を使って、次のようなデータを挿入してみることにします。

id product_name price stock_quantity
1 桐の棒 1,200 10
2 鉄の斧 16,000 8

解説 03

ここをクリックすれば解説が表示されます。

まずは単一レコードの挿入について考えてみましょう。単一レコードの挿入は次のように書きます。 

INSERT INTO `products`
  (`id`, `product_name`, `price`, `stock_quantity`)
VALUES
  (NULL, '桐の棒', 1200, 10);

なるほど。入力する列名を括弧で括って提示して、その後、対応する値たちを VALUES の後に括弧で括って書いていくんじゃな。

そうです。テーブル作成時と同様、列名はバッククォート（ ` ）かダブルクォート（ " ）で囲うか・もしくは囲わないことが可能です。値は文字列のみシングルクォート（ '  ）で囲います。文字列の値は列名と違って、必ずシングルクォート（ ' ）で囲ってください。

そもそも id 列は主キーで自動採番なのに、いちいち NULL とかよく分からないものを指定しなくちゃいかんのか？

いいえ、省略できます。

INSERT INTO `products`
  (`product_name`, `price`, `stock_quantity`)
VALUES
  ('桐の棒', 1200, 10);

ところで、 id 列を省略しない場合には NULL を指定していたという点について何か思い出しませんか？

PRAGMA 文でメタ情報を見た時に、主キーなのに NOT NULL 制約がオフになっていたことかな。

はい。よく覚えていましたね。ここで主キーなのに NOT NULL 制約がオフになっていたことが利いてきています。……実は INSERT 文は次のように列名の列挙を省略することもできます。 

INSERT INTO `products`
VALUES
  (NULL, '桐の棒', 1200, 10);

ほう。この場合はさすがに id 列は省略できんよな？

はい、できません。必ず登録した列順に対応する値全てを並べる必要があります。

まあ当然じゃろな。

さらに INSERT 文は、次のようにすることで複数レコードを一気に挿入していくこともできます。

INSERT INTO `products`
  (`product_name`, `price`, `stock_quantity`)
VALUES
  ('桐の棒', 1200, 10),
  ('鉄の斧', 16000, 8);

この場合も列名の列挙は省略できるのかな？

はい。次のようにできます。

INSERT INTO `products`
VALUES
  (NULL, '桐の棒', 1200, 10),
  (NULL, '鉄の斧', 16000, 8);

なるほど。

ただし、列名を省略した書き方はテーブル構造に依存するため、実務ではあまりおすすめできません。

04. ドワーフ・ガンツのデータ読み込み

相変わらずデータ挿入しただけじゃ、中がどうなってるのか分からんのじゃな。正常終了は沈黙じゃったな。

そうです。今度は CRUD の Read の登場です。

READ 文じゃなくて、 SELECT 文なんじゃったな。

とりあえず、入力したデータ全件を読み込んでみましょう。

id product_name price stock_quantity
1 桐の棒 1,200 10
2 鉄の斧 16,000 8

解説 04

ここをクリックすれば解説が表示されます。

SELECT 文はデータベースを扱う際に最も利用する SQL 文です。全レコード全列読み込むには、次のように書きます。

SELECT
  `id`, `product_name`, `price`, `stock_quantity`
FROM
  `products`;

INSERT 文に比べてシンプルじゃな。

はい。ワイルドカード（ * ）を使って列名を省略すれば、もっとシンプルに書けます。ワイルドカードを使った場合は、強制的にすべての列が読み込み対象になります。

SELECT
  *
FROM
  `products`;

なるほど。全列のデータが必要な場合はワイルドカードを使い、必要な列が限定されている場合は列指定の書き方をすれば良さそうじゃな。

05. ドワーフ・ガンツのデータ更新

実はな、テーブルデータにあった「鉄の斧」はな、あれは違うんじゃ。あれは鉄メッキをした金の斧なんじゃ。値段も 2,000,000 円でな。

そ、そうなんですか？ どこかの泉のインチキ女神からでももらったんですか？

想像に任せる。更新は CRUD の Update じゃな？

データを更新する場合、どのデータを更新したいのかを必ず指定してやる必要があります。テーブル products の場合、レコードを一意に指定できる id 列で指定してやると良いでしょう。

id product_name price stock_quantity
1 桐の棒 1,200 10
2 鉄の斧 16,000 8

つまりこのテーブルの場合、 id が 2 のレコードを指定するんじゃな？

解説 05

ここをクリックすれば解説が表示されます。

まずどのレコードを更新するのか指定せずに更新する場合は UPDATE 文で次のように書きます。

UPDATE `products`
  SET `product_name`='金の斧', `price`=2000000;

これはどのレコードを更新すればいいのか分からなくてエラーになるのかな？

違います。全てのレコードが更新されます。

ひぇっ！ つまり全商品「金の斧」になっちゃうのか？

そうです。条件を指定するためには WHERE 句を使います。

UPDATE `products`
SET
  `product_name`='金の斧', `price`=2000000

…………ワシに spell_name 列がおかしいって言わせたいんじゃろ？

言わないんですか？

集約関数を使った際に集約されない列が任意の値になったのと同じで、グループ化されなかった列は該当するグループの中から任意の値が表示されるんじゃろ？

察しがいいですね。

おまえさんといると用心深くなるわ。

ちなみに element_name 列もグループ化された後にグループ内の任意の値が表示されているわけですが、そもそも element_name でグループ化しているので、各グループのラベルのように使えるわけです。 GROUP BY した場合 SELECT 句ではグループ化した列と集約関数だけを使うと覚えておけばよいでしょう。ですから、ここでも spell_name 列のようなものは SELECT 句に入れるべきではありません。

なるほど。……これが前回言っていた注意事項、集約関数は抽出されたレコードの指定された列の全値を集計するんじゃなくて、グループごとに集計するとか言っていたやつじゃな？

はい。その通りです。属性ごとの在庫額を集計するには GROUP BY 句でグループ化した上で、列の掛け算と SUM() を使います。集計列は total_price としておきます。

SELECT
    `element_name`, SUM(`price`*`stock_quantity`) AS `total_price`
  FROM `material_spell_elements_view`
  GROUP BY `element_name`;

element_name total_price
水 3,900
火 24,137,000
雷 97,500

おまけですが、集計結果に対して条件検索をかけるには HAVING 句を使います。これで total_price が 10,000 円以上の列のみ表示できるようになります。

SELECT
    `element_name`, SUM(`price`*`stock_quantity`) AS `total_price`
  FROM `material_spell_elements_view`
  GROUP BY `element_name`
  HAVING SUM(`price`*`stock_quantity`)>=10000;
material_spell_elements_view
element_name total_price
火 24,137,000
雷 97,500

ん？ WHERE 句じゃダメなのか？

ダメですね。 WHERE 句も加えると FROM 句・ WHERE 句・ GROUP BY 句・ HAVING 句・ SELECT 句という順に実行されるので、グループ化された後のレコードに対しては WHERE 句で抽出できません。 WHERE 句はレコードに対する条件、 HAVING はグループに対する条件と覚えておけばよいでしょう。

ほぉん。

CASE 式は値を返すので SELECT 句内で試してみましょう。

SELECT
  CASE
    WHEN '検証したい値'='ピッピロピー' THEN 'すぐ合致'
    WHEN '検証したい値'='検証したい値' THEN '次で合致'
    ELSE '合致しなかった'
  END;

出力結果です。

CASE
次で合致

んー。なんか急に複雑じゃな。

まず CASE 式は CASE と END で全体を囲います。 WHEN で条件を設定し、いくつでも並べることができます。 WHEN のいずれかに合致したら THEN の後の値が返ります。どの条件にも合致しなければ ELSE の後の値が返ります。

つまりこういうことじゃな。

CASE
  WHEN 条件 THEN 値
  WHEN 条件 THEN 値
  ELSE 値
END;

はい。条件は順に見ていって最初に条件を満たしたときに値を返します。

つまり最初の WHEN で条件が満たされていたとき、次以降の WHEN と ELSE は無視されるということじゃな。

はい。あと、さっきの例のような場合、こんな風にも書けます。

SELECT
  CASE '検証したい値'
    WHEN 'ピッピロピー' THEN 'すぐ合致'
    WHEN '検証したい値' THEN '次で合致'
    ELSE '合致しなかった'
  END;

CASE '検証したい値'
次で合致

列名を AS 句で変えられないのか？

SELECT
  CASE '検証したい値'
    WHEN 'ピッピロピー' THEN 'すぐ合致'
    WHEN '検証したい値' THEN '次で合致'
    ELSE '合致しなかった'
  END AS `case_test`;
case_test
次で合致

    `product_name`, `price`,
